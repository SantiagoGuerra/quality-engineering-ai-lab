# Playwright Agent plan: idempotent candidate retry

Generated by Claude Code `playwright-test-planner` in read-only mode on 2026-07-21. Human review note: the product uses an `Idempotency-Key`; phone is not part of the candidate schema, and an E2E should prefer API setup for a lost-response condition.

## Core scenarios

1. Happy path: recruiter submits valid synthetic name/email/location; one result and success state.
2. Lost response: allow the first server request to complete but withhold/abort its browser response, retry the identical payload and key, then verify exactly one persisted candidate by API/list.
3. Genuine first 500: retry creates exactly one item and surfaces a recoverable state.
4. Same email with changed payload: expect duplicate conflict rather than a silent second record.
5. Rapid double submit: saving state disables additional submission; exactly one item.
6. Verify uniqueness using filter by the synthetic email.
7. TTL boundary: document expected idempotency retention before automating.

## Success and failure

- Success: one persisted candidate per logical intent; no crash, hang, silent loss or misleading duplicate error.
- Failure: two records, unrecoverable UI error, or behavior inconsistent with the documented idempotency contract.

## Human decisions required

- Confirm idempotency retention/TTL before scenario 7.
- Keep deterministic API/Hurl assertions as gate; this agent plan is guidance only.
- Never weaken an assertion or baseline in response to a failure.
